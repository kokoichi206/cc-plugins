# ヒアリング設計スキル

`$ARGUMENTS` の案件概要/docs を読み、見積もり精度を上げるために顧客へ聞くべき質問を網羅的に生成し、ファイルに出力する。

## 引数

- `$ARGUMENTS`: 案件概要のテキスト、または docs のディレクトリ/ファイル。未指定なら対象を質問する。

## 手順

### 1. 不確実性の高い領域を洗い出す

`$ARGUMENTS` を Read し、見積もりの振れ幅を生む領域を特定する。最低限、以下の観点を網羅する。

- スコープ (対象範囲・優先度・フェーズ分け)
- 非機能要件 (性能・可用性・セキュリティ・対応端末/ブラウザ)
- データ (件数・移行の有無・初期データ・個人情報の扱い)
- 外部連携 (連携先システム・API 仕様の有無・認証方式)
- 権限 (ロール設計・アクセス制御の粒度)
- 運用・保守 (リリース後の運用体制・監視・SLA・保守範囲)

### 2. 各領域の質問を網羅する

- 各領域について、回答が見積もりに直結する具体的な質問を作る。
- 抽象的な質問 (例: 「要件はありますか」) は避け、回答で工数が動く粒度にする (例: 「同時アクセスのピークは何ユーザーを想定しますか」)。

### 3. 「これで決まること」を併記

- 質問ごとに「この回答で決まること = どの不確実性/振れ幅が縮むか」を 1 行で添える。
- 可能なら、関係する WBS タスクや見積もりレンジへの影響も書く。

### 4. 優先度付け

- 見積もりへの影響が大きい質問から並べる、または HIGH/MEDIUM/LOW を付ける。
- 影響が大きく未確定の項目を冒頭で目立たせる。

### 5. ファイル出力

- 領域ごとに見出しを分け、質問 + 「これで決まること」 + 優先度の表/箇条書きでファイルに Write する (例: `docs/hearing-questions.md`、または指定の出力先)。出力先が曖昧なら確認する。

## 注意事項

- CLAUDE.md のコーディング/対話規約に従う
- 抽象質問でなく、回答が見積もりに直結する具体的な質問にする
- 各質問に「これで決まること」を必ず併記する
- 案件固有の機密が含まれる場合は出力ファイル内に留め、チャットに垂れ流さない
- 全角括弧は使わず半角記号を優先する
- 探索には fd/rg 相当 (Glob/Grep) を使い、作業スコープを越えない
