# 工数・費用見積もりスキル

`$ARGUMENTS` で指定された docs/会話/リポジトリから、工数 (振れ幅付き) と費用を見積もり、結果をファイルに出力する。

## 引数

- `$ARGUMENTS`: 見積もり対象の docs ディレクトリ/ファイル、またはリポジトリパス。会話で共有された要件も対象に含めてよい。
- `--with-demo`: 「デモ版まで」と「本実装まで」を明確に分けて集計する (指定が無くても両方出すのが望ましい)。

## 手順

### 1. 要件の把握

- `$ARGUMENTS` の docs を Glob で列挙し Read する。会話で共有された要件もあわせて整理する。
- 機能要件と非機能要件を分けて箇条書きにする。
- 不明な前提 (対象範囲・対応端末・性能目標・既存資産の有無など) は、勝手に埋めず「不明点」として書き出し、必要なら質問する。仮定で進める場合は「仮定:」と明示する。

### 2. コード規模の把握 (リポジトリがある場合)

- 既存リポジトリが対象なら規模感を掴む。コミット履歴は `git log --oneline` 等で確認する。
- ファイル数/主要モジュールの把握には Glob/Grep を使う。
- 新規開発で既存コードが無い場合はこのステップをスキップし、その旨を前提に明記する。

### 3. WBS でタスク分解

- 要件をタスク (WBS) に分解する。設計/実装/テスト/レビュー/環境構築/ドキュメントなど、開発以外の作業も漏らさない。
- 各タスクが「デモ版に含まれるか」「本実装で必要か」を区別できるよう印を付ける。

### 4. 工数の見積もり (振れ幅付き)

- 各タスクに 楽観 / 標準 / 悲観 の 3 点で工数を置く (人日 or 人時、単位を明記)。
- 不確実性の高いタスクほど楽観と悲観の幅を広げる。
- **AI 併用前提の係数**を明示する。AI Agent 併用で短縮が見込めるタスク種別 (定型実装・テスト雛形・調査など) と、短縮が効きにくいタスク (要件調整・外部連携の擦り合わせなど) を分け、適用した係数 (例: 実装系 0.6 倍) と根拠を書く。

### 5. 集計

- `デモ版まで` と `本実装まで` を分けて、楽観/標準/悲観それぞれの合計工数を出す。
- AI 係数 適用前/適用後の両方を併記し、効果が分かるようにする。

### 6. 費用の算出

- 単価 × 工数 で費用を出す。単価が共有されていなければ勝手に置かず、確認するか「単価未確定 (要入力)」のプレースホルダで構造だけ作る。
- デモ版/本実装、AI 係数 適用前/適用後で費用も併記する。

### 7. 前提・除外条件とファイル出力

- 見積もりの前提、除外条件 (スコープ外)、リスク、そして **いつ時点の見積もりか (基準日)** を必ず明記する。
- 結果をファイルに Write する (例: `docs/estimate-<基準日>.md`、または指定の出力先)。出力先が曖昧なら確認する。
- 単価などの機密はファイル内に留め、チャットには合計レンジ程度に留める。

## 注意事項

- CLAUDE.md のコーディング/対話規約に従う
- 不明な前提を勝手に埋めない。質問するか「仮定:」として明示する (知ったかぶり禁止)
- いつ時点の見積もりか (基準日) を必ず記載する
- AI 併用前提の係数は適用箇所・倍率・根拠をセットで示す
- 単価などの機密は出力ファイル内のみに留め、チャットに垂れ流さない
- 全角括弧は使わず半角記号を優先する
- 探索には fd/rg 相当 (Glob/Grep) を使い、作業スコープを越えない
