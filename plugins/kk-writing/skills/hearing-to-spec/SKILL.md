# ヒアリング to 仕様確定スキル

`$ARGUMENTS` で指定されたヒアリングメモから、決定事項と未決事項を整理し、仕様確定ドキュメントを生成する。

## 引数

- `$ARGUMENTS`: ヒアリングメモのパスと、任意の出力先
  - 第 1 引数: ヒアリングメモのファイル/ディレクトリ
  - 第 2 引数 (任意): 出力先パス (例: `docs/todos/`)。未指定なら出力先を確認する

## 手順

### 1. 論点を抽出する

- ヒアリングメモを Read し、論点・要望・制約を洗い出す
- メモに書かれた事実と、解釈・提案を区別して把握する

### 2. 決定事項 / 未決事項に分類する

- 合意済み・確定しているものを「決定事項」に
- 確定していない・検討が必要なものを「未決事項」に
- 決定と未決を混在させず、明確に分ける

### 3. 未決事項に候補とおすすめを付ける

各未決事項について以下を記載する:

- 候補: 取り得る選択肢を複数挙げる
- おすすめ: どれを推すか
- 理由: なぜおすすめか (根拠を必ず添える)

### 4. 既存ドキュメントとの差分を指摘する

- 既存の docs を Glob/Grep で照合する
- 「どのファイルの・どの記述が・どう変わるか」を具体的に明示する
- 影響範囲が不明な場合は仮定として明示する

### 5. 仕様確定ドキュメントを出力する

- 指定された出力先 (例: `docs/todos/` 配下) にファイルを生成する
- 構成例:

```markdown
# 〇〇 仕様確定メモ

## 決定事項

- ...

## 未決事項

### 論点 A

- 候補: 案1 / 案2
- おすすめ: 案1
- 理由: ...

## 既存ドキュメントへの影響

- `docs/spec.md` の 〇〇 が △△ に変わる
```

## 注意事項

- CLAUDE.md のコーディング/対話規約に従う
- 決定事項と未決事項を混在させない
- おすすめには根拠を必ず添える
- 事実 (メモに書かれた内容) と提案 (こちらの判断) を分離する
- 全角括弧を使わず半角記号を優先する
