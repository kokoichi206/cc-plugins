# 引き継ぎ README 生成スキル

`$ARGUMENTS` で指定された対象から、引き継ぎ・他社提供用の README / 手順書を生成する。

## 引数

- `$ARGUMENTS`: 対象ディレクトリのパスと、任意のオプション
  - `--audience external`: 他社・社外向け。社内固有の情報や認証情報を除外する
  - `--audience internal`: 社内・後任者向け。社内固有の前提を残してよい
  - 対象ディレクトリ未指定: カレントディレクトリを対象とする

## 手順

### 1. 対象を把握する

- ディレクトリ構成を Glob で確認する
- `package.json` などのマニフェストを Read し、`scripts` / 依存 / 実行コマンドを把握する
- 必要に応じて `git log` で直近の変更や未完了の文脈を確認する

### 2. 章立てを設計する

以下を基本構成とし、対象に存在しない章は省く:

- 概要 (何をするものか)
- 前提 (必要なランタイム・ツール・バージョン)
- セットアップ (依存インストールなど)
- 実行 (ローカルでの起動・動作確認)
- デプロイ (デプロイ手順・対象環境)
- 環境変数 (名前と用途の一覧)
- 残タスク (未完了・既知の課題)

### 3. external 指定時のフィルタリング

- `--audience external` の場合、社内固有のホスト名・社内ツール名・URL・認証情報・個人名などを除外する
- 除外して意味が通らなくなる箇所は、汎用的な表現に置き換えるか、不明点として明示する

### 4. 陳腐化対策

- 逐次変わる手順の冗長なコピーは避け、要点とコマンドに絞る
- 詳細はソース (`package.json` の `scripts` など) を参照する形にし、二重管理を避ける

### 5. README に出力する

- 対象ディレクトリに `README.md` を生成する。既存がある場合は Edit で章を追記・更新する
- 環境変数は名前と用途のみを表で記載し、値は書かない:

```markdown
| 環境変数       | 用途                     |
| -------------- | ------------------------ |
| `DATABASE_URL` | 接続先データベースの URL |
```

- 残タスクは箇条書きで、何が残っているか明確にする:

```markdown
## 残タスク

- [ ] 〇〇のエラーハンドリング追加
- [ ] 本番デプロイ手順の検証
```

## 注意事項

- CLAUDE.md のコーディング/対話規約に従う
- 環境変数は名前と用途のみ記載し、値は絶対に書かない
- 残タスクは箇条書きで明確にする
- 全角括弧を使わず半角記号を優先する
- 陳腐化しやすい冗長なコピーは書かず、要点に絞る
