# 明細・領収書 突合スキル

`$ARGUMENTS` で指定された明細 CSV (クレジットカード/銀行/Amazon 等) と領収書を突合し、立替・経費の対応関係を作る。「明細にあるが領収書なし」「領収書あるが明細なし」「自分の立替か他人の支払いか」を分類し、精算漏れ・取り違えを検出する。

## 引数

- `$ARGUMENTS`: 突合対象の明細 CSV ファイルと、領収書を格納したディレクトリ。複数の明細 (カード/銀行/Amazon) を渡してよい。指定が曖昧な場合は対象を確認してから進める。

## 手順

### 1. 入力の収集と確認

1. 明細 CSV を Read で読み、各 CSV の列構成 (日付/金額/摘要/利用者など) を把握する。フォーマットがカード/銀行/Amazon で異なる前提で扱う。
2. 領収書ディレクトリを Glob で列挙する (PDF/画像/CSV など)。ファイル名や中身から日付・金額・店名の手掛かりを抽出する。
3. **どの明細が誰の口座/カードか** を最初に確認する。自分名義か他人名義かが不明なものは「不明」として保持し、勝手に自分の立替へ寄せない。
4. 通貨・税込/税抜・日付フォーマットの揺れを確認し、正規化方針を決める。

### 2. 突合 (マッチング)

1. 日付・金額・摘要をキーに、明細レコードと領収書を突き合わせる。
2. 完全一致を優先し、次に「金額一致 + 日付が数日以内」などの近接一致を候補とする。近接一致は確定ではなく **候補** として扱う。
3. 1 対多 (合算決済) や多対 1 (分割) の可能性も考慮する。自動で 1 対 1 に丸めない。
4. 件数が多い・集計が必要な場合は、簡単な `python3` スクリプトを書いて突合・集計してよい (標準ライブラリの csv 等)。スクリプトは入力を読み取り専用で扱い、結果は別ファイルに出力する。

### 3. 区分の判定

各レコードに以下の区分を付与する。判断できないものは空欄にせず「不明」と明示する。

- 用途区分: 立替 / 経費 / 対象外
- 所有者区分: 自分 / 他人 / 不明

所有者の判定は厳密に行う。自分のクレジットカード以外の支払いが混ざる前提で、名義・カード末尾・口座などの根拠と突き合わせる。根拠が無いものを自分の立替に分類しない。

### 4. 不一致の洗い出し

- 明細にあるが領収書なし (証憑不足の候補)
- 領収書あるが明細なし (未計上/別口座の候補)
- 金額や日付がずれて一致しきらないもの (要確認)
- 所有者が他人または不明のもの (取り違え/精算対象外の候補)

それぞれ件数と内訳を一覧化する。

### 5. 結果の出力

1. 突合結果を表形式でまとめ、CSV として **別ファイル** に出力する (例: `reconcile_result.csv`, `unmatched.csv`)。出力先は作業ディレクトリ配下とする。
2. 各行に「根拠 (どの領収書/明細と対応)」「区分」「確度 (確定/候補/不明)」を持たせる。
3. チャットには件数サマリーと未解決の確認事項のみを示し、個別の金額や口座情報は垂れ流さずファイルへ書く。

### 6. 確認の依頼

候補・不明として残ったものを列挙し、ユーザーに区分の確定を依頼する。特に「これは本当に自分の立替か」を 1 件ずつ確認できる形で提示する。

## 注意事項

- CLAUDE.md のコーディング/対話規約に従う
- 自分のクレジットカード以外の支払いが混ざる前提で、所有者判定を厳密に行う。根拠なく自分の立替へ寄せない
- 金額を概算・四捨五入で誤魔化さない。一致しない差分は差分として明示する
- 推測で埋めない。不明な金額・所有者・用途は「不明」と明記し、勝手に確定しない
- 元データ (明細 CSV・領収書) は変更しない。出力は必ず別ファイルに書き出す
- 突合の取りこぼし・取り違えが疑わしい箇所はエラーを握りつぶさず明示する
- 突合ロジックや所有者判定で詰まったら o3 MCP に相談する
