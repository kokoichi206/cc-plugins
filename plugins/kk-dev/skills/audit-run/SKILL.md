# 監査オーケストレーター

プロジェクトに対して包括的な監査を実行し、結果を GitHub Issue として報告する。

## 概要

5 つの専門監査エージェントを並列実行し、結果を集約して GitHub Issue を作成する。

## 監査エージェント

1. **セキュリティ監査**: 認証/認可、入力検証、機密情報管理、データ保護
2. **アーキテクチャ監査**: 設計ドキュメントとの整合性、レイヤー分離、命名規則
3. **ベストプラクティス監査**: TypeScript/React/フレームワークのベストプラクティス
4. **UI/UX 監査**: アクセシビリティ、コンポーネント設計、スタイリング一貫性
5. **依存関係監査**: 脆弱性、古いパッケージ、未使用依存関係

## 実行手順

### Step 1: 準備

1. 現在の日付を取得 (YYYY-MM-DD 形式)
2. 現在のブランチの最新コミットハッシュを取得
3. 出力ディレクトリを作成: `docs/audits/{date}/`

### Step 2: 監査の並列実行

Task tool を使用して、5 つの監査エージェントを **並列** で実行する。

各エージェントには以下を指示:

- まず `CLAUDE.md` と `package.json` を読み、プロジェクトの技術スタックと規約を把握する
- 該当する監査領域について網羅的にチェックする
- 出力先: `docs/audits/{date}/{type}.md`

**重要**: 5 つの Task を **同時に** 発行すること (並列実行)。

各エージェントへの指示は以下の通り:

#### Task 1: セキュリティ監査

OWASP Top 10 およびプロジェクト固有のセキュリティリスクを検出する。

チェック項目:

- 認証/認可: 認証設定、セッション管理、認証ミドルウェアの適用範囲、Server Actions/API Routes での認証チェック漏れ
- 入力検証: バリデーション有無、SQL インジェクション対策、XSS 対策
- 機密情報管理: 環境変数の適切な使用、ハードコードされた機密情報、クライアントへの漏洩チェック、ログへの機密情報出力
- データ保護: トークンの暗号化保存、個人情報の取り扱い、Cookie のセキュリティ属性
- 依存関係の脆弱性: `pnpm audit` / `npm audit` の結果
- エラーハンドリング: スタックトレースの露出、エラーメッセージでの情報漏洩

出力: `docs/audits/{date}/security.md` (スコア X/10、重要度別の件数、具体的な検出項目と修正案)

#### Task 2: アーキテクチャ監査

設計ドキュメント (CLAUDE.md 等) との整合性を検証する。

チェック項目:

- レイヤードアーキテクチャの整合性: 依存方向の違反、循環参照
- ディレクトリ構造: 設計に沿った構成か、server/client の分離
- 命名規則: ファイル名、コンポーネント、関数、DB テーブル/カラム
- インポート規則: パスエイリアスの使用、相対参照の制限
- API 設計: Server Actions 優先の方針
- DB 設計: マイグレーション、スキーマ定義の整合性

出力: `docs/audits/{date}/architecture.md` (スコア X/10、重大/軽微/推奨の件数)

#### Task 3: ベストプラクティス監査

業界標準のベストプラクティスへの準拠を検証する。

チェック項目:

- TypeScript: `any` 型、`as` キャスト、型ガード、strict モード
- React: Hooks ルール、再レンダリング、メモ化、キー
- フレームワーク: Server/Client Components の分離、Metadata、Loading/Error 境界
- ORM: N+1 クエリ、トランザクション、不要なカラム取得
- エラーハンドリング: 一貫したパターン、空の catch
- パフォーマンス: バンドルサイズ、画像最適化、遅延読み込み
- テスト: カバレッジ、命名規則、モック

出力: `docs/audits/{date}/best-practices.md` (スコア X/10、カテゴリ別スコア)

#### Task 4: UI/UX 監査

UI コンポーネントの品質を検証する。

チェック項目:

- アクセシビリティ: セマンティック HTML、ARIA 属性、キーボードナビゲーション、フォーカス管理、色コントラスト、alt テキスト
- コンポーネント設計: 単一責任、Props 設計、再利用性
- スタイリング一貫性: CSS フレームワークの一貫した使用、レスポンシブデザイン
- フォーム設計: バリデーションフィードバック、エラーメッセージ、ローディング状態
- インタラクション: ローディングインジケーター、エラー/空状態、確認ダイアログ

出力: `docs/audits/{date}/ui.md` (スコア X/10、重大/改善推奨/軽微の件数)

#### Task 5: 依存関係監査

パッケージ依存関係の健全性を検証する。

チェック項目:

- セキュリティ脆弱性: `pnpm audit` / `npm audit`、既知の CVE
- 古い依存関係: `pnpm outdated` / `npm outdated`、deprecated パッケージ
- 未使用依存関係: 使用されていないパッケージの検出
- ライセンス: ライセンス互換性
- バンドルサイズ: 大きすぎるパッケージ、軽量な代替

出力: `docs/audits/{date}/deps.md` (スコア X/10、カテゴリ別件数)

### Step 3: 結果の集約

各監査レポートからスコアとサマリーを抽出し、総合レポートを作成する。

### Step 4: GitHub Issue の作成

[Issue テンプレートとコメント追加手順](reference/issue-template.md) に従って Issue を作成する。

### Step 5: 完了報告

- 作成した Issue の URL を表示
- 全体のスコアサマリーを表示

## 引数

$ARGUMENTS

引数で以下を指定可能:

- 空の場合: 今日の日付を使用
- 日付指定 (YYYY-MM-DD): 指定した日付を使用
- `--dry-run`: GitHub Issue を作成せず、ローカルにレポートのみ生成

## 注意事項

- 実行には数分かかる場合がある
- GitHub CLI (`gh`) が認証済みであること
