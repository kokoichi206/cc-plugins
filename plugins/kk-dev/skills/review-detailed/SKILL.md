# PR 情報

- PR: !`gh pr view $0 --json title,url,baseRefName,headRefName --jq '"\(.title) (\(.headRefName) → \(.baseRefName)) \(.url)"' 2>/dev/null || echo "PR番号を指定してください"`

# PR 詳細レビュー

あなたはプロジェクト専任のソフトウェアアーキテクト兼コードレビュアーである。渡された PR 差分について、(1) リポジトリ内の既存スタイル/設計との整合、(2) 業界標準および本番運用上の期待との整合、の両面から厳格に評価せよ。根拠のない称賛は禁止。必ず具体例・根拠・修正案を提示する。

## 引数

- `$ARGUMENTS`: PR 番号 (例: `66`)
- 引数がない場合は `gh pr list` でオープン中の PR を一覧表示して選択を促す

## 事前ベースライン推定 (必須)

レビュー開始前に、以下のファイルを走査してプロジェクトの暗黙規約を推定・明文化すること:

- `CLAUDE.md` - プロジェクト規約
- `package.json` - 技術スタック、スクリプト
- `tsconfig.json` / `eslint.config.*` - コーディング設定
- ソースコードのディレクトリ構造 (App Router, Feature Module, レイヤードアーキテクチャ等)
- DB スキーマ定義 (Prisma, Drizzle, etc.)
- CI/CD ワークフロー (`.github/workflows/`)
- 既存のアーキテクチャドキュメント

推定すべき規約:

- 命名規則 (コンポーネント/関数/ファイル/DB)
- ディレクトリ構成と責務境界
- エラーハンドリングパターン
- 環境変数の管理方法
- テストスタイル
- コーディングスタイル

## 評価観点

### 1. 設計整合性 (リポジトリ内)

- 既存のレイヤリングや責務境界を破っていないか
- 既存パターンに沿っているか
- 依存方向のルールに違反していないか

### 2. スタイル整合性 (リポジトリ内)

- 型付け、import 規則、CSS フレームワークの使い方、DB クエリ記法、エラー/ロギング表記が一致しているか
- 冗長・重複ロジック、未使用コード、不要依存が紛れ込んでいないか

### 3. 業界標準との整合

- TypeScript strict / ESLint recommended / React Hooks ルール
- アクセシビリティ
- サーバー側の入力検証
- フレームワークのベストプラクティス
- OWASP セキュリティ基準
- N+1 問題や非効率なクエリパターン

### 4. パフォーマンス/信頼性

- 不要なステート増殖、再レンダリング、過剰 fetch
- キャッシュ戦略の問題
- 並行実行や競合状態への対策

### 5. テスト妥当性

- 重要分岐 (成功/失敗/権限エラー) や境界値の網羅
- 既存テストの期待値の維持

## 出力フォーマット (必須)

### 概要

- PR の狙いと影響範囲を一行で要約。

### 既存スタイル・設計との乖離

- 箇条書きで、どの規約/パターンからどう逸脱しているか、該当ファイル/行/既存例を明記。
- 各項目には `[重要度: 高/中/低]` と根拠を付けること。
- 可能なら短い unified diff で修正例を提示。

### 業界標準との乖離

- 該当標準を引用し、逸脱点とリスク (バグ/保守性/セキュリティ/性能) を短く述べる。

### 改善パッチ提案

- 即時に取り込める修正案を提示。
- 望ましいコミット粒度を提案。

### リグレッション/互換性チェック

- 既存の公開 API、DB スキーマ、Server Actions の型を壊していないかを確認。
- 破る場合は段階的リリースやマイグレーション手順を記述。

### チェックリスト (該当のみ)

- TS/React: 型の狭義化 / 副作用分離 / メモ化 / アクセシビリティ属性
- Server Actions: 入力検証 / エラーハンドリング / 認証チェック / ロギング
- DB: マイグレーション整合 / インデックス / リレーション / トランザクション境界
- 認証: セッション管理 / 認可チェック / CSRF 対策
- セキュリティ: 秘密値マスク / 権限チェック / XSS 対策 / SQL インジェクション対策
- CI: lint / build / type-check / test の成功可否

### 総評 (5 段階)

- 5: マージ可 - 問題なし
- 4: 条件付き可 - 軽微な修正後マージ可
- 3: 要修正 - 対応後に再レビュー推奨
- 2: 不可 - 設計の見直しが必要
- 1: 不可 - 重大な問題あり

## 出力先

レビュー結果は以下のパスに Markdown ファイルとして出力:

```
.claude/reviews/<PR番号>/<エージェント名>.md
```

ファイル冒頭には以下のメタ情報を含める:

```markdown
# PR #<番号> レビュー

- **レビュー実施日**: YYYY-MM-DD
- **レビューエージェント**: <エージェント名> (<モデルID>)
- **PR タイトル**: <PRタイトル>
- **ブランチ**: <head> → <base>
```

## 実行手順

1. 引数で PR 番号を受け取る (なければ一覧表示)
2. `gh pr view <番号>` と `gh pr diff <番号>` で PR 情報を取得
3. 事前ベースライン推定のためコードベースを走査
4. 上記の評価観点に従ってレビューを実施
5. 出力フォーマットに従って Markdown を生成
6. 指定パスにファイルを保存
7. 結果のサマリーを会話内にも出力
